# PBL③ テスト・品質保証 テスト仕様書 ＆ 不具合分析レポート

対象アプリケーション：一人暮らし向け献立管理・提案アプリ

本ドキュメントは、作成されたFlaskアプリケーションに対し、品質向上を目的として「SHIFTの3視点（仕様視点・実装視点・利用視点）」から網羅的に洗い出したテストケース、および現状のソースコードから検出された潜在バグとその再現手順をまとめたものです。GitHub Desktop等での管理や、AIとの共同デバッグ用のリファレンスとして活用してください。

## 2. 現状コードから見つかった具体的な不具合（バグ）と再現手順

### 【不具合①】仕様・利用視点：記録される日付が「2026-06-25」に完全固定されている
* **現象：** どの日に「この献立を食べて記録する」を押しても、履歴には必ず「2026-06-25」と記録されてしまい、今日の日付や自由な日付で管理できない。
* **再現手順：**
  1. アプリを起動し、任意の条件で「献立を提案してもらう」をクリック。
  2. 「この献立を食べて記録する」をクリック。
  3. 画面下の食事履歴を確認すると、日付がすべて「2026-06-25」になっている。
* **原因箇所：** `index.html` の 56行目付近：`<input type="hidden" name="date" value="2026-06-25">`

### 【不具合②】実装視点：予算が空欄のときにValueErrorでクラッシュする
* **現象：** 予算入力欄の値を消して空欄にしたり、ブラウザのデベロッパーツール等で文字列を送信すると、サーバー側で `ValueError` が発生し、画面に「Internal Server Error」が表示される。
* **再現手順：**
  1. ブラウザで予算の「400」を完全に削除して空欄にする。
  2. 「献立を提案してもらう」ボタンを押す。
  3. 画面が真っ白になり、Flaskのデバッグ画面または500エラーが発生する。
* **原因箇所：** `app.py` の 85行目：`budget = int(request.form.get("budget", 400))` （空文字 `""` は int() に変換できないため）

### 【不具合③】利用視点：予算が少なすぎて提案に失敗した際、ユーザーに何も伝わらない
* **現象：** 予算を「100円」など極端に低く設定して検索すると、内部ロジックは `None` を返すが、画面には「条件を入力してボタンを押してください」という初期メッセージが表示され、検索が失敗したのかまだ押していないのか区別がつかない。
* **再現手順：**
  1. 予算を「100」に変更し、ジャンルを「和食」にしてボタンを押す。
  2. 画面の右側が初期状態（「条件を入力して...」）のまま変化しない。
* **原因箇所：** `index.html` の `{% if suggested_menu %}` の `{% else %}` 句の判定ロジック不足。

### 【不具合④】実装視点：レシピCSV内の価格データに不正な値があるとアプリが起動・稼働できない
* **現象：** `recipes.csv` の価格（price）列に「時価」や「500円」のように文字列が含まれていると、データ読み込み時に `ValueError` でシステムが停止する。
* **再現手順：**
  1. `recipes.csv` のいずれかの行の price を「300」から「未定」に変更する。
  2. アプリを起動し、トップページにアクセス（または提案ボタンを押す）。
  3. 型変換エラーが発生し、画面が表示されない。

  ### 【要望・新不具合⑤】仕様・利用視点：食材の幅が狭く、健康管理（栄養バランス）の機能が不足している
* **現象：** 提案される献立が価格とジャンルだけで選ばれるため、食材が偏りやすく、一人暮らしのユーザーが一番気にする「栄養バランス」や「何が足りないか」が一切わからない。
* **再現手順（テスト方法）：**
  1. アプリで「和食・2品」の提案を複数回実行する。
  2. 提案されたメニューを見ても、カロリーや栄養素（タンパク質、脂質など）の記載がない。
  3. これまでに食べた履歴（meals.csv）を見ても、自分に何の栄養素が不足しているのかを可視化（グラフやアラート表示など）する仕組みがない。
* **原因：** `recipes.csv` に栄養素のデータ項目がなく、Python側の提案ロジックやHTML側の表示機能にも栄養計算・可視化の仕組みが実装されていないため。
